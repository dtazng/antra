# Implementation Plan: Go Containerized Backend with PostgreSQL

**Branch**: `015-go-backend` | **Date**: 2026-03-15 | **Spec**: [spec.md](spec.md)

## Summary

Replace the existing AWS Lambda + DynamoDB backend with a Go modular monolith containerized in Docker, backed by PostgreSQL. The system exposes a versioned REST API (`/v1`) for auth, sync, persons, logs, follow-ups, notifications, and settings. A separate worker process (same codebase, `CMD` override) runs periodic jobs for follow-up scheduling and push notification delivery using `robfig/cron`. Stack: chi router, pgx/v5 + sqlc for database access, goose for migrations, golang-jwt for JWT, argon2id for passwords, firebase-admin-go for push.

**Performance Goals**: p95 < 200ms; < 100 concurrent users

## Key Design Decisions

### Authentication flow

1. Register: argon2id hash → INSERT user + user_settings → INSERT refresh_token → return JWT + refresh UUID
2. Login: lookup by email → verify argon2id → INSERT refresh_token → return JWT + refresh UUID
3. Refresh: lookup refresh_token by UUID → check expiry → return new JWT
4. Logout: DELETE refresh_token row
5. Auth middleware: extract Bearer → `jwt.Parse` HS256 → inject `userID` into `context.Context`

### Sync design

- **Push**: iterate changes per entity, call service SyncUpsert/SyncDelete; check `server.updated_at > client.updated_at`; collect conflicts; single transaction commit; return summary
- **Pull**: `SELECT ... WHERE user_id=$1 AND updated_at > $2 ORDER BY updated_at LIMIT $3` (includes tombstones)
- Server always sets `updated_at = NOW()` on accepted records (canonical timestamp)

### Worker process

```
scheduler.go sets up:
  @every 5m → follow_up_job.CheckDueFollowUps(db)
  @every 5m → notification_job.DispatchNotifications(db, pushClient)
Graceful shutdown via OS signal.
```

### Follow-up job

```sql
UPDATE follow_ups
SET status = 'due', updated_at = NOW()
WHERE status IN ('pending', 'snoozed')
  AND due_date <= CURRENT_DATE
  AND (snoozed_until IS NULL OR snoozed_until <= CURRENT_DATE)
  AND deleted_at IS NULL
```
Then for each newly-due follow-up: INSERT notification if user.notifications_enabled = true.

### Notification job

```sql
SELECT * FROM notifications
WHERE status = 'scheduled'
   OR (status = 'failed' AND retry_count < max_retries)
```
For each: fetch active device tokens → firebase SendEachForMulticast → INSERT delivery records → UPDATE notification status.

## Deployment Recommendation

**Single-server deployment** (Fly.io, Railway, or any Docker VPS):
- `antra-api` container: `./antra api` (or `CMD ["./antra", "api"]`)
- `antra-worker` container: `./antra worker`
- PostgreSQL: managed (Neon, Supabase, or self-hosted)
- Secrets via environment variables (DATABASE_URL, JWT_SECRET_KEY, FIREBASE_CREDENTIALS_JSON)

No Kubernetes, no load balancer needed at solo-founder scale.

## Risks and Trade-offs

| Risk | Impact | Mitigation |
|------|--------|-----------|
| sqlc regeneration required after every schema change | Medium | `make sqlc` in Makefile; document in README |
| argon2id memory parameter may OOM on small containers | Low | Default to OWASP minimums (64MB); document in .env.example |
| firebase-admin-go requires valid credentials file | Medium | Graceful degradation: log error, skip push, continue |
| Single-node PostgreSQL (no HA) | Low at this scale | Managed DB service handles backups + failover |
| robfig/cron crash stops all jobs | Low | Container restart policy; structured logging on panic |
