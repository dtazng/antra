# Implementation Plan: Task Lifecycle & Review Flow

**Branch**: `002-task-lifecycle` | **Date**: 2026-03-10 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-task-lifecycle/spec.md`

---

## Summary

Implement a full task lifecycle system for Antra: daily carry-over of yesterday's unfinished tasks, a task detail view with complete lifecycle history, and a weekly review queue for tasks older than 7 days. Tasks are mutated in place (no duplication). A new append-only `task_lifecycle_events` table tracks every state transition. The existing `bullets` table gains four new columns. A new `TaskLifecycleService` owns all state machine logic.

## Constitution Check

### Principle I: Code Quality ✅

- `TaskLifecycleService` is a pure Dart class (no Flutter imports) — single responsibility, injectable, testable
- All task state transitions route through the service exclusively — no UI code writes status directly
- `TaskLifecycleDao` handles all DB queries; service delegates to it
- Derived display states are computed functions, not stored — no risk of staleness

**Note**: The existing `migrateBullet()` in `BulletsDao` is deprecated but not deleted (existing migrated rows must remain readable). It is marked `@Deprecated` and excluded from all new call sites.

### Principle II: Testing Standards ✅

- Unit tests required for all `TaskLifecycleService` methods (happy path + all edge cases from spec)
- Unit tests required for `watchCarryOverTasks` and `watchWeeklyReviewTasks` query correctness
- Tests use drift in-memory database — no SQL mocking
- Offline path: all operations are local-only; no network dependency in this feature

### Principle III: UX Consistency ✅

- Quick actions use `showModalBottomSheet` — consistent with existing `CreatePersonSheet` pattern
- Cancel action includes 3-second undo snackbar — satisfies "destructive actions require confirmation" without a blocking modal
- All three sections have proper empty states
- Carry-over section appears passively below today's entries — capture flow is not blocked

**Tension addressed**: "From Yesterday" could feel like productivity pressure. Mitigated by calm visual treatment (no red badges, no urgency copy), placement below today's capture bar, and no notifications triggered by carry-over.

### Principle IV: Performance ✅

- Carry-over query uses indexed `day_logs.date` + new `idx_bullets_created_at` index
- Weekly review query uses `idx_bullets_created_at` (added in migration)
- Both queries return `Stream<List<T>>` from drift — reactive, no polling
- Task detail loads lifecycle events via single indexed query — no N+1

### Privacy & Data Integrity ✅

- `task_lifecycle_events` is local-only (sync deferred to future spec)
- Events are append-only — no silent rewrites
- `carry_over_count` is stored for UI display; event log is the source of truth if they ever diverge

## Key Design Decisions

### 1. No task duplication — dayId is mutable

"Keep for Today" updates `bullets.day_id` to today's day_log, preserving `created_at`. Replaces the old `migrateBullet()` pattern that created a copy.

### 2. Mutual exclusion via created_at threshold

- Tasks `created_at` within 7 days, in yesterday's day_log → **"From Yesterday"**
- Tasks `created_at` older than 7 days, status active → **Weekly Review**

Enforced at the DAO query level. UI trusts the streams.

### 3. Backward-compatible status values

New `TaskStatus` constants map to existing DB strings. Only `'backlog'` is new. `'migrated'` is deprecated — never written by new code, still readable for existing rows.

### 4. Lifecycle events — local-only in v1

`task_lifecycle_events` is NOT synced in this feature. Sync integration is deferred to avoid scope creep. Events are purely local history.

### 5. Cancel with 3-second undo

Satisfies constitution Principle III "destructive actions require confirmation" without a blocking dialog. Immediate soft-delete with snackbar undo window. After 3 seconds: committed.
